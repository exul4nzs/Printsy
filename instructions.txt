# 🎨 Agentic AI Prompt: Custom Print Studio – Clean & Expandable

You are an AI coding agent. Build a **production‑ready, mobile‑first Custom Print Studio** that allows customers to design and order **custom‑printed products** – starting with **clothing (T‑shirts, shorts, hoodies)** but architected to easily add **tarpaulins, DTF transfers, stickers, mugs, and other print services** in the future.

## 🧱 Tech Stack (Fixed)

| Layer | Technology |
|-------|------------|
| Backend | Django 5.x + Django REST Framework |
| Frontend | Next.js 14 (App Router) + TypeScript + Tailwind CSS |
| Database | PostgreSQL (dev: SQLite ok) |
| Payments | Stripe Checkout (with GCash for Philippines) |
| Storage | AWS S3 or local `media/` (for MVP) |
| Canvas Editor | Fabric.js (web) – touch‑friendly, responsive |

## 🎨 Visual & UX Theme: "Clean, Minimalist & Product‑Focused"

Implement the frontend with this exact aesthetic:

- **Color Palette**: Neutral base (off‑white, warm grays, soft beiges) with **one vibrant accent** (e.g., orange or teal) for CTAs and key interactive elements. Avoid clutter.
- **Typography**: Use `'Inter'` for body text and UI elements. Use a bold geometric sans‑serif for the logo and main headings.
- **Layout & Spacing**: Generous whitespace, modular grid‑based layout (Bento Box style). Cards have consistent, subtle rounded corners and shadows.
- **Mobile‑First**: All screens responsive. Touch targets appropriately sized. Gestures (pinch, drag, rotate) smooth on canvas.
- **Trust Signals**: Clear pricing, simple checkout steps, secure payment badges.

## 🎯 Core MVP Scope (Clothing First)

Build a fully functional **clothing customization MVP** with the following features. The code must be structured so that **adding new product types (tarpaulin, DTF, etc.) requires minimal changes** – i.e., use a polymorphic product model, dynamic configuration UI, and extensible order handling.

### 1. Product Catalog (Extensible)
- **Abstract `Product` model** with common fields: `name`, `base_price`, `is_active`, `product_type` (choices: `clothing`, `tarpaulin`, `dtf`, etc.), `thumbnail`, `created_at`.
- **`ClothingVariant` model** (for now) – size, color, stock quantity. Later, other product types can have their own variant models or use JSON attributes.
- **Admin interface** to add/edit products and variants.

### 2. Customer Frontend (Next.js + Tailwind)
- **Product Gallery** – displays all active products. Use the defined aesthetic: clean grid, generous whitespace, product cards with image, name, price, and a prominent “Customize Now” button (accent color).
- **Product Detail / Configurator** – dynamic based on type:
  - For `clothing`: **Fabric.js 2D editor** with:
    - Garment mockup image (loaded from product).
    - **Layout**: Two‑column on desktop (canvas left, tools right). On mobile: full‑screen canvas with collapsible bottom sheet for tools.
    - Toolbar: simple, clean icons for 'Add Text', 'Add Image', 'Color Picker', 'Undo', 'Redo', 'Save Design'.
    - Add text: dialog to input text, choose font (small dropdown), select color.
    - Add clipart (predefined library of 10+ SVGs).
    - Upload custom image (PNG/JPG).
    - Undo/redo, clear canvas.
    - Export PNG (transparent background) and save design configuration (canvas JSON).
  - For future types (tarpaulin, DTF): placeholder message or simple file upload + dimension fields – not required for MVP but the code must allow easy extension (e.g., a `configuratorComponent` mapping).
- **Cart** (Zustand or Context) – stores product variant, quantity, design configuration (if any), unit price.
- **Checkout** – collect customer name, address, phone. Integrate Stripe Checkout with `payment_method_types=['card', 'gcash']`. Use a clean, centered modal or dedicated page.
- **Order Confirmation** – display order summary, design preview, order status.

### 3. Backend (Django + DRF)
- **Models**:
  - `Product`, `ClothingVariant` (FK to Product, with size, color, stock).
  - `CustomDesign` – stores `product` (FK), `design_config` (JSON – Fabric canvas data), `preview_image` (S3 URL), `created_at`.
  - `Order` – customer info, items (JSON list: `product_id`, `variant_id`, `design_id`, `quantity`, `price`), total, `stripe_payment_intent_id`, status (`pending`/`paid`/`processing`/`shipped`), `created_at`.
- **API Endpoints**:
  - `GET /api/products/?type=clothing` – list products (filter by type).
  - `GET /api/products/<id>/variants/` – list available variants.
  - `POST /api/designs/` – save design, return design ID and preview URL.
  - `POST /api/create-payment-intent/` – create Stripe PaymentIntent.
  - `POST /api/webhooks/stripe/` – confirm payment, mark order paid.
  - `GET /api/orders/` (admin only) – list orders with download links.
  - `PATCH /api/orders/<id>/` – update order status.
- **Admin**: Django admin for managing products, variants, designs, orders.

### 4. Extensibility Requirements (Important!)
- **Product type registry** – In the frontend, have a mapping like:
  ```ts
  const configuratorMap = {
    clothing: ClothingConfigurator,
    tarpaulin: TarpaulinConfigurator, // not implemented, but placeholder
    dtf: DtfConfigurator,
  };
  ```
- **Backend** – Use Django's `ContentType` framework or a simple `product_type` field to allow different product logic in the future. For MVP, only `clothing` is fully implemented, but the code should not break if `tarpaulin` is added later.
- **Order item JSON** – Store enough metadata (product type, design config reference, dimensions, etc.) so that future production workflows can read it.

## 🧪 Example Flow (Clothing)
1. User visits `http://localhost:3000` → sees T‑shirt, shorts, hoodie (clean grid).
2. Clicks on T‑shirt → opens Fabric.js editor (two‑column or bottom sheet).
3. Adds text, clipart, saves design.
4. Adds to cart, selects size/color.
5. Checkout → Stripe (test GCash payment).
6. Order stored; admin can download PNG from Django admin.
